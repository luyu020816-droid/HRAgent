Spring AI Alibaba 集成 DashScope 超时问题排查与解决文档
一、问题现象
在使用 Spring AI Alibaba 框架集成阿里云 DashScope（通义千问）时，调用 AI 接口出现间歇性超时：
超时错误：java.net.SocketTimeoutException: timeout
错误频率：大部分请求超时，偶尔成功
错误信息：
text
org.springframework.web.client.ResourceAccessException: I/O error on POST request
for "https://dashscope.aliyuncs.com/api/v1/services/aigc/text-generation/generation": timeout
二、问题排查过程
2.1 初步诊断（排除法）
排查项 方法 结果 结论
网络连通性 ping dashscope.aliyuncs.com ✅ 通 网络正常
API 有效性 curl 命令测试 ✅ 200 OK，响应 304ms API Key 和服务都正常
配置超时 application.yml 配置 ❌ 仍然超时 配置文件不生效
SDK 配置 添加 dashscope-sdk-java 依赖 + Constants 配置 ❌ 仍然超时 框架未使用该配置
2.2 关键发现
curl 测试成功的关键信息：

bash
< HTTP/1.1 200 OK
< x-dashscope-timeout: 298 # 服务端实际处理时间仅 298ms
{"output":{"text":"你好！很高兴见到你..."}}
这说明：
服务端响应很快（< 300ms）
超时是客户端主动放弃，而非服务端慢
2.3 根本原因分析
架构层面：
text
Spring AI Alibaba DashScopeChatModel
↓ 使用
RestClient (框架内部创建)
↓ 使用
OkHttp3ClientHttpRequestFactory
↓ 使用
OkHttpClient (使用默认超时配置：10秒)
问题本质：
Spring AI Alibaba 框架内部自己创建了 OkHttpClient 实例
该实例使用 OkHttp 的默认超时配置（连接/读取/写入各 10 秒）
开发者通过 application.yml 或 Constants 配置的超时参数无法注入到框架内部的客户端
当网络抖动或响应略慢时（>10秒），客户端主动断开连接，导致超时
为什么是"偶发性"超时？
正常情况：响应在 10 秒内返回 → 成功
网络波动/服务端负载高时：响应超过 10 秒 → 超时失败

三、最终有效方案
通过自定义 RestClient.Builder Bean，利用 Spring Boot 的自动配置覆盖机制：
关键点 说明
@Primary 确保自定义 Bean 优先级高于框架内部 Bean
ClientHttpRequestFactorySettings Spring Boot 3.x 提供的 HTTP 客户端配置工具
readTimeout: 300秒 大模型响应可能较慢，需要足够长的读取超时
